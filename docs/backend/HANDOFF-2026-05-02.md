# Handoff · Estado del backend a 2026-05-02

> Documento del **estado actual** de Byvaro tras la fase de migración
> a Supabase + cierre de fugas cross-tenant. Léelo de arriba a abajo
> antes de tocar nada · es la foto consolidada de qué está hecho, qué
> queda y dónde están los puntos críticos.

## 1 · Resumen ejecutivo

**Estado**: Byvaro arrancó como prototipo frontend Vite/React con
`localStorage` como source-of-truth de TODO. En las últimas iteraciones:

1. **Supabase es la única fuente de verdad** · 60+ tablas con RLS
   multi-tenant · cliente público con anon key + JWT · cero
   `service_role_key` en frontend.
2. **`localStorage` eliminado completamente** · sustituido por
   `memCache` (in-memory Map) que se hidrata desde Supabase al login.
   Sign-out vacía la memoria · cero residuos. **Excepción**:
   `sessionStorage` para `accountType`/`currentUser` (rol activo
   per-tab) y `localStorage` que **maneja Supabase internamente**
   para sus tokens de auth.
3. **Tenant isolation hardened** · 4 capas de defensa cross-empresa:
   RLS por fila + column-level grants + `organization_private_data`
   con RLS member-only + `private_metadata` en `promotions` con
   función SECURITY DEFINER que sólo devuelve a owner/colaboradores
   activos.

**No hay backend custom todavía** · Supabase actúa como BaaS completo.
Cuando se requiera un backend Node/Python custom (p.ej. para
integraciones Stripe, Firmafy, Resend), se enchufa por encima de
Supabase Postgres.

## 2 · Cómo arrancar el sistema en otro servidor

### 2.1 · Variables de entorno requeridas

```env
# .env.local (frontend · expuestas en build)
VITE_SUPABASE_URL=https://<project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=eyJh...                    # public · safe en bundle

# .env.local (scripts · NUNCA exponer al cliente)
SUPABASE_DB_URL=postgresql://postgres.<ref>:<password>@aws-0-eu-west-1.pooler.supabase.com:5432/postgres
SUPABASE_SERVICE_ROLE_KEY=eyJh...                 # ⚠️ server/scripts only
```

### 2.2 · Aplicar migraciones SQL en orden

Las migraciones viven en `supabase/migrations/`. Aplica en orden
cronológico (los timestamps lo garantizan):

```bash
# Idempotentes · puedes re-correr sin romper nada
for f in supabase/migrations/*.sql; do
  psql "$SUPABASE_DB_URL" -f "$f"
done
```

Las migraciones clave que aplican el patrón actual:
- `20260501000000_localstorage_migration.sql` · primera tanda de
  tablas (user_settings, org_settings, agency_cartera,
  whatsapp_settings, dismissed_invitations, etc.)
- `20260501100000_split_plan_packs.sql` · plan dual (agency_pack +
  promoter_pack)
- `20260501110000_fix_agency_signup_kind.sql` · backfill agency
  signup_kind
- `20260502000000_no_more_localstorage.sql` · invitations,
  favorite_agencies, agency_licenses, agency_doc_requests, user_2fa,
  microsite_templates, org_email_domains
- **`20260502120000_tenant_isolation_hardening.sql`** ⭐ · cierre de
  fugas cross-tenant. Mueve datos sensibles (tax_id, comisiones
  default, marketing intel, main_contact_*) a
  `organization_private_data` con RLS member-only · split de
  `promotions.metadata` en `metadata` público + `private_metadata`
  con column-level grant + RPC SECURITY DEFINER

## 3 · Arquitectura de datos

### 3.1 · Capas

```
┌─────────────────────────────────────────────────┐
│  React components / pages                       │
│  · Hooks síncronos                              │
└────────────┬────────────────────────────────────┘
             │ import { useFooHook } from "@/lib/foo"
             ▼
┌─────────────────────────────────────────────────┐
│  Helpers en src/lib/*.ts                        │
│  · Frontera estable · API drop-in               │
│  · Cache reads via memCache                     │
│  · Mutations con write-through a Supabase       │
└────────┬────────────────────────────┬───────────┘
         │                            │
         │ memCache.getItem            │ supabase.from(table).upsert(...)
         │ memCache.setItem            │
         ▼                            ▼
┌──────────────────┐         ┌──────────────────────┐
│  memCache        │         │  Supabase REST       │
│  (in-memory Map) │         │  · RLS por fila      │
│  · Render sync   │         │  · Column grants     │
│  · Vacío al      │         │  · Bridge views api.*│
│    sign-out      │         │  · SECURITY DEFINER  │
└──────────────────┘         └──────────┬───────────┘
                                        │
                                        ▼
                             ┌──────────────────────┐
                             │  Postgres            │
                             │  · is_org_member()   │
                             │  · is_org_admin()    │
                             │  · public + api      │
                             └──────────────────────┘
```

### 3.2 · Flujo al login

```
LOGIN (Supabase auth)
  ↓
AuthStateChange "SIGNED_IN"
  ↓
SupabaseHydrator.tsx → splash bloqueante "Cargando workspace…"
  ↓
hydrateAll() · Promise.all de 12 hidratadores en paralelo:
  · supabaseHydrate.ts (orgs + profiles + offices + private_data)
  · seedHydrator.ts (promotions + units + leads + sales + registros + private_metadata RPC)
  · invitaciones, sentEmails, promoCollabStatus, agencyLicenses,
    companyEvents, favoriteAgencies, agencyOnboarding, agencyProfile,
    twoFactor, developerPacks
  ↓
memCache poblado · splash desaparece
  ↓
React hooks renderizan instantáneo (read sync de memCache)
```

### 3.3 · Flujo en una mutación

```
Component → useXHook().save(patch)
  ↓
helper.saveX(patch):
  1. memCache.setItem(KEY, JSON.stringify(next))
  2. window.dispatchEvent(CustomEvent("byvaro:x-changed"))
  3. void syncXToSupabase(next)         ← write-through async
  ↓
Otros componentes re-renderizan vía evento
  ↓
syncXToSupabase llama a supabase.from(table).upsert(...)
  · RLS valida is_org_member o equivalente
  · Si falla, queda solo en memoria (best-effort)
  · Phase futura · retry/queue para garantía
```

### 3.4 · Flujo al sign-out

```
SIGNED_OUT
  ↓
clearSupabaseCache() + clearMemCache()
  ↓
memoria vacía · ningún dato residual
  ↓
JWT supabase descartado → próxima query da 401
```

## 4 · Tenant isolation · 4 capas de defensa

Ver `docs/backend/architecture/tenant-isolation.md` para detalle.
TL;DR:

1. **RLS por fila** · cada tabla con `organization_id` filtra por
   `is_org_member()`. Tablas dual-tenant (sales, registros, leads)
   filtran por `organization_id` OR `agency_organization_id`.

2. **Column-level grants** · en `promotions`, los GRANTs de SELECT
   excluyen `private_metadata`. Anon/authenticated NO pueden leerla
   directamente · solo via RPC.

3. **Tabla privada** · `organization_private_data` (1:1 con
   organizations) alberga tax_id, comisiones default, marketing
   intel, main_contact_*. RLS = `is_org_member(organization_id)`.
   Cross-tenant queries devuelven 0 rows.

4. **RPCs SECURITY DEFINER** · cuando el frontend necesita datos
   privados de promociones donde es participante (owner o colaborador
   activo), llama a `api.list_promotion_private_metadata()` que
   internamente comprueba membership y filtra. NO expone datos a
   no-participantes.

### 4.1 · Verificación rápida

```bash
psql "$SUPABASE_DB_URL" -c "
SET ROLE anon;
SELECT count(*) FROM public.organization_private_data;  -- 0 esperado
SELECT count(tax_id) FROM public.organizations;          -- 0 esperado
SELECT private_metadata FROM public.promotions LIMIT 1;  -- permission denied
RESET ROLE;
"
```

## 8 · Reglas duras invariantes (CLAUDE.md)

Lee `CLAUDE.md` entero antes de tocar nada. Las reglas críticas:

1. **Backend acoplado** · cero `supabase.from()` en componentes.
   Toda lectura/escritura va por un helper de `src/lib/*`.
2. **Cero `localStorage` en componentes** · usa `memCache` o el
   helper canónico (`useUserSetting`, `useOrgSetting`).
3. **Multi-tenant por defecto** · cada tabla nueva con
   `organization_id` + RLS desde día 1.
4. **Public refs canónicas** · `PR#####` (promo), `UN########`
   (unit), `RG#########` (registro), `CO#######` (contact),
   `US#######` (user), `ID######` (tenant). Generadas via FNV-1a
   determinístico (seedRef) o random (generatePublicRef).
5. **Datos sensibles privados** · cualquier dato comercial (tax_id,
   comisiones, marketing intel, contactos internos) va a
   `organization_private_data` o `private_metadata`. NUNCA en
   tablas con SELECT public.

1. `npx tsc --noEmit` · type check

3. Comprueba env vars (especialmente `VITE_SUPABASE_*`)
